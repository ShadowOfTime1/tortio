# Tortio Roadmap

План развития проекта. Обновляется по мере необходимости.

## Текущее состояние (v1.17.0)

**Что есть:**
- Flutter-приложение для масштабирования рецептов тортов (Android primary, iOS/desktop поддерживаются базово).
- Локальное хранение через `SharedPreferences` (без сервера).
- Auto-update через GitHub Releases (приложение само скачивает APK).
- Полноценный UI: рецепты, секции, ингредиенты, фото, теги, drag-and-drop, темы, статистика, импорт/экспорт.
- Безопасность: ProGuard/R8, signed APK, portrait lock.
- CI: `flutter analyze` + `flutter test` на каждый push/PR.

**Что НЕТ:**
- Бэкенда / облачной синхронизации.
- Авторизации.
- Кросс-устройственной работы (рецепты в одной установке = в одном телефоне).
- Multi-tier тортов (один диаметр на рецепт).
- Английской версии.
- PDF экспорта.
- Play Store / F-Droid распространения.

## Phase 2 — Backend + Авторизация (~3–5 недель)

Ключевой архитектурный шаг. Все «защиты от копирования», синхронизация между устройствами, шаринг рецептов между пользователями — становятся возможны только когда есть бэкенд.

### 2.1 Выбор стека (1–2 дня обсуждений)

**Опции:**

| Вариант | Плюсы | Минусы | Цена |
|---|---|---|---|
| **Supabase** | Готовый auth + Postgres + storage + realtime, open source, есть Flutter SDK | Vendor lock-in (мягкий — open source) | Free tier до ~50K MAU, потом ~$25/мес |
| **Firebase** | Самый зрелый Flutter-стек, Google Auth искаропки | Vendor lock-in (жёсткий), NoSQL (Firestore) | Free tier щедрый, потом per-usage |
| **Своя БД + API** (Node/Go/Rust + Postgres) | Полный контроль, легко мигрировать | Сильно больше работы, нужна инфра, hosting | VPS от $5/мес + время |
| **Pocketbase** | Single-binary BaaS, SQLite/Postgres, простой | Молодой проект | Free, нужен self-host |

**Рекомендация:** Supabase. Минимум работы, профессиональный auth искаропки, Postgres (нормальная реляционка), есть row-level security. Можно слезть позже если понадобится.

### 2.2 Дизайн данных
- [ ] Схема БД: `users`, `recipes`, `recipe_tiers`, `recipe_sections`, `ingredients`, `tags`, `custom_section_types`, `recipe_versions` (history).
- [ ] Решить: каждый рецепт принадлежит одному пользователю; шаринг — через explicit invite или link?
- [ ] Решить: что синхронизируется (рецепты, custom types, settings?), что остаётся локальным.

### 2.3 Auth flow
- [ ] Регистрация: email + пароль + verification email.
- [ ] Логин: email/пароль; reset password по email.
- [ ] (опционально) OAuth — Google/Apple sign-in.
- [ ] Token storage на устройстве — `flutter_secure_storage` (Keystore/Keychain), НЕ SharedPreferences.
- [ ] Session refresh, logout, удаление аккаунта.
- [ ] Privacy Policy (юридическое требование при регистрации).

### 2.4 Sync strategy
- [ ] Локальный режим работает как раньше (без аккаунта — просто recipes в SharedPreferences).
- [ ] При логине — миграция локальных рецептов в облако (с подтверждением).
- [ ] Конфликт-резолюшен: last-write-wins по `updated_at` per-recipe.
- [ ] Offline-first: правки локально → синхронизация когда появится сеть.

### 2.5 Защита от использования копий
Когда есть backend, можно реально:
- [ ] Server-side rate limiting на login.
- [ ] **Play Integrity API** — backend отказывает копиям, не подписанным нашим ключом.
- [ ] App attestation для iOS (когда дойдём).
- [ ] Никаких секретов в коде клиента — всё через backend proxy.

## Что НЕ делать

- ❌ **Прятать код на GitHub** до того, как есть backend. Без backend — security through obscurity, бесполезно.
- ❌ **Эмбедить ключи API в код клиента.** Никаких "секретных" токенов в Dart-коде, даже в private репо. Через backend proxy.
- ❌ **Хешировать пароли на клиенте.** Это работа backend (`bcrypt`/`argon2` server-side).
- ❌ **Использовать `SharedPreferences` для токенов после auth.** Только `flutter_secure_storage`.
- ❌ **Делать кастомный auth с нуля.** Supabase/Firebase/Auth0 — все сделают лучше нас.
- ❌ **Раздавать APK через мессенджеры/diскорды** после переезда в Play Store. Создаёт fragmented user base.

## Ближайшее решение

**На что тратим следующую сессию (твой выбор):**

🥇 **Multi-tier торты** — большой архитектурный апдейт. Высокая ценность для свадебных тортов, после этого приложение станет «полным».

🥈 **Несколько мелких улучшений** — поиск по ингредиентам, bulk-delete, PDF экспорт. Меньше риска, больше быстрых побед.

🥉 **Подготовка к backend** — обсудить архитектуру, выбрать стек, нарисовать схему БД. Без кода — только дизайн на бумаге, чтобы потом быстро реализовать.

🎯 **Английская локализация** — если планируешь делиться вне русскоязычного круга.
